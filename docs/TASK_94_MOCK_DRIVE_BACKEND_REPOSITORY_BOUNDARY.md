# TASK-94 Mock Drive Backend Repository Boundary

## Purpose

TASK-94 implements the first SALMAN OS Drive file hub repository boundary before any real Google Drive API integration.

This task connects the current Drive-style file hub UI to a mock repository result shape so the later actual Google Drive adapter can replace the mock implementation behind the same boundary.

## Scope

Implemented:

- Drive file domain types in `src/domain/driveFiles.ts`
- `DriveFileRepository` interface in `src/data/repositories/driveFileRepository.ts`
- `driveFileMockRepository` in `src/data/adapters/mock/driveFileMockRepository.ts`
- `useClientDriveFiles` hook for UI loading/error/result state
- `FilesPanel` changed to consume repository result metadata instead of local folder matching
- repository selection now includes the Drive file repository
- mock repository tests for listing, filtering, default exclusion rules, safe diagnostics, and file lookup

Not implemented:

- Google Drive API calls
- Google SDK package usage
- OAuth, service account, token, credential, or `.env.local` loading
- Supabase schema, RLS, RPC, SQL execution, or `drive_*` table activation
- embedding, vector, RAG answer generation, or chatbot UI
- real file upload, rename, move, sync, or archive mutation

## Boundary Structure

```text
FilesPanel
  |
useClientDriveFiles(clientId)
  |
driveFileRepository
  |
driveFileMockRepository
  |
sanitized mock Drive metadata
```

The current runtime uses only the mock repository. Supabase repository selection keeps the same mock Drive repository because no actual Drive metadata source is active yet.

## Domain Shape

The mock boundary exposes:

- `DriveFileSummary`
- `DriveFileCategory`
- `DriveFileSource`
- `DriveFileSensitivity`
- `DriveFileStatus`
- `DriveFileRepositoryResult`
- `DriveFileDiagnostic`

Default list behavior:

- returns active files only
- excludes `archived` files unless `includeArchived` is explicitly requested
- always excludes `excluded` restricted/do-not-index files from file hub results
- returns sanitized diagnostics only
- returns empty `sourceUrl` values instead of real Google Drive links

## Future Actual Adapter Point

The future actual adapter should implement `DriveFileRepository` with the same methods:

- `listDriveFiles(input)`
- `getDriveFileById(input)`
- `listDriveFileCategories(input)`

The later adapter must stay server-side and must not expose Google credentials, access tokens, service account JSON, or raw file contents to the browser.

## Security Boundary

TASK-94 intentionally does not read:

- `.env.local`
- credential files
- OAuth token files
- service account JSON
- private keys
- Google Drive file contents

TASK-94 also does not add any `VITE_` credential environment variable or service role key usage.

## Test Coverage

Added `tests/mock/driveFileRepository.test.ts` to verify:

- active mock Drive files are returned through the repository boundary
- category/status/sensitivity filters work
- archived and restricted files are excluded from the default list
- diagnostics are safe and do not expose credential/token/service account metadata
- `getDriveFileById` does not widen archived access by default
- category counts are generated by the repository boundary

Existing activation gate tests were updated so `RepositorySet` includes the Drive repository slot.
